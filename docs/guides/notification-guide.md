# task-relay 通知文面ガイド

出典: [personas.md](../../personas.md) §3.1, §4.0, §4.1, §7.2, §7.4, §8.1, §8.5, §10

本書は通知の文面と重要度分離を扱う。
通知の配送実装、Webhook 詳細、rate limit 制御の正本ではない。

## 1. 目的

通知は「情報を増やす」のではなく「人間の判断を短縮する」ために送る。
特に P1 と P2 に対して、スマホ上で次の 2 点を短時間で判定できることを目的とする。

1. 今すぐ介入すべきか。
2. 介入するなら何をすべきか。

## 2. 基本原則

- 進捗通知より介入通知を優先する。
- 正常進行を逐一知らせない。異常、承認待ち、停止、コスト異常だけを主対象にする。
- 1 通目で `何が起きたか / 今すぐ動くべきか / 次の行動` が分かるようにする。
- 局所障害と全体障害を文面レベルで混同させない。
- Discord は判定と最小介入に寄せ、詳細説明は Forgejo や log に逃がす。

補足:
- 本書 §3 に出てくる `human_review_required`, `needs_fix`, `system_degraded`, `breaker open` は内部分類の説明用語である。
- ユーザー向け文面では enum 名をそのまま出さず、判断語彙へ翻訳する。

## 3. 通知クラス

### 3.1 受理通知

用途:
- task 受付成功
- task 受付失敗

原則:
- DM は使わない。
- 内容は短く、受理可否と追跡先だけを示す。
- 成功時は安心を、失敗時は再投入要否を示す。

最低限の要素:
- task_id
- 受理結果
- 追跡先

### 3.2 介入通知

用途:
- `human_review_required`
- `needs_fix`
- 範囲外変更や review 不通過
- 手動承認待ち

原則:
- 何が止まったかを明記する。
- 自動再試行に委ねない理由を短く添える。
- 次の手動行動を 1 つに絞って示す。

最低限の要素:
- task_id
- 現在 state
- 停止理由の短い要約
- 次の行動
- PC で見るべき場所

### 3.3 全体障害通知

用途:
- `system_degraded`
- breaker open
- dispatch stop

原則:
- task 単位の失敗と混同させない。
- 「個別 task を見に行く前に全体停止を理解すべき」文面にする。
- 深夜対応を想定し、待機可能かどうかを先に伝える。

最低限の要素:
- 全体障害であること
- 影響範囲
- dispatch 停止か task 停止か
- 人間が今やるべきこと
- 朝まで待ってよい条件、または即介入が必要な理由

内部分類の読み替え:
- `system_degraded` は「全体障害」として表示する。
- breaker open は「全体保護中」または「新規開始を一時停止中」として表示する。
- breaker open は全体影響ではあるが、`system_degraded` と同じ文言にしてはならない。

### 3.4 コスト / rate 通知

用途:
- 利用料の異常増加
- rate limit 接近または枯渇

状態:
- v1.0 では概念設計のみを置く。
- threshold、計算 window、通知頻度、通知チャネルの最終確定は次版へ送る。
- したがって本節は P1 要求の将来受け皿であり、現在の実装正本ではない。

原則:
- `いま危険か / 近いうちに危険か` を分ける。
- 単なる数値ではなく、行動判断に使える形で伝える。
- P1 が「今日は回し続けてよいか」を判断できることを優先する。

最低限の要素:
- 現在値
- しきい値との関係
- 推奨行動

## 4. 文面テンプレート

### 4.1 受理成功

`task {task_id} accepted. 追跡先: {url}`

### 4.2 介入通知

`task {task_id} is waiting for human action. 理由: {short_reason}. 次: {next_action}.`

### 4.3 全体障害通知

`system degraded. {impact}. dispatch is stopped. 次: {next_action}.`

### 4.4 コスト / rate 警告

`usage alert. {current_status}. 推奨: {next_action}.`

補足:
- このテンプレートは将来版の叩き台であり、v1.0 実装必須ではない。

テンプレート運用ルール:
- 先頭 1 文は 80 文字程度で収める。
- state 名は必要時のみ露出し、内部語彙だけで完結させない。
- 詳細理由は 1 通目に詰め込みすぎず、追跡先へ逃がす。
- `human_review_required`, `needs_fix`, `system_degraded`, `breaker open` をそのまま通知本文の主語にしない。

## 5. 書いてはいけない文面

- 進捗だけを長々と流す通知
- `error occurred` のように行動につながらない通知
- 局所障害と全体障害が読み分けできない通知
- 1 通に複数 task の行動要求を詰め込む通知
- 深夜対応で複数画面遷移を前提にした通知

## 6. 受け入れ基準

- P1 はスマホだけで「放置可 / 要介入」を判定できる。
- P2 は深夜 1 分以内に「止める / 待つ / 再開準備」を選べる。
- P3 が後から読んだとき、通知が局所障害か全体障害か誤読しない。
- 通知だけでは詳細が足りなくても、必ず追跡先がある。
