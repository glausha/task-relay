# task-relay モバイル UX 原則

出典: [personas.md](../../personas.md) §3.1, §4.1, §7.1, §7.2, §7.3, §8.5, §10

本書は Discord を中心としたモバイル利用時の判断体験を定義する。
Web UI の実装詳細や画面遷移仕様ではなく、情報設計の原則を扱う。

## 1. 前提

P1 と P2 はモバイルでは深掘りしない。
モバイルの役割は `判定` と `最小介入` であり、`精査` と `長時間レビュー` は PC に寄せる。

## 2. Surface ごとの役割

### 2.1 Discord

使う場面:
- task 投入
- 要点確認
- 停止
- 承認
- 再試行の起点

Discord に載せるべきもの:
- task の現在地
- 今すぐの危険有無
- 次に押すべき操作

P2 の `1 画面で判別する` 要件に対する主 surface:
- Discord の `/status` 応答を主とする。
- `/status` は少なくとも `局所停止 / 全体障害 / 全体保護中(dispatch pause)` を明示ラベルで返す。

Discord に載せすぎないもの:
- 長い履歴
- 差分全文
- 詳細ログ
- 複雑な復旧手順

### 2.2 Forgejo / runner-cli

使う場面:
- 履歴精査
- 差分確認
- 長文レビュー
- 復旧判断の最終確定
- 保守作業

## 3. モバイル画面で即判定できるべき情報

- task_id
- 現在 state
- 正常進行か介入要か
- 局所障害か全体障害か
- 次の推奨行動

補足:
- `state 名が読める` だけでは不足である。
- 画面上では、state から人間の判断に必要な意味へ翻訳する。
- `/status` や通知文面では enum 名そのものではなく、判断語彙へ変換したラベルを表示する。

## 4. モバイルで許容する操作

- task 投入
- `approve`
- `cancel`
- `retry`
- 状態確認

制約:
- 1 回の操作で大きな副作用を持つものは、確認文または PC 側導線を伴う。
- 深夜対応では 2–3 アクションで終わる範囲を超えない。
- 強制操作は少数に保ち、意味の重なりを作らない。

## 5. 情報の並べ方

- 先頭で危険度を示す。
- 次に「何が起きたか」を 1 行で示す。
- 最後に「次に何をするか」を示す。

望ましい順序:

1. 危険度または重要度
2. task / system の対象
3. 現在地
4. 次の行動
5. 追跡先

## 6. モバイル UX の禁止事項

- 正常進捗の垂れ流し
- 専門用語だけで意味を理解させる
- 複数 task の文脈を 1 通に混在させる
- PC が無いと危険度判定すらできない設計
- 深夜に長文コマンド入力を強いる設計

## 7. 受け入れ基準

- P1 は 30 秒以内に「今どうなっているか」を把握できる。
- P2 は 1 分以内に「寝てよい / 今動く」を判定できる。
- 詳細が必要なときだけ PC に移る構造になっている。
- モバイル上で見える文面だけで局所障害と全体障害を誤読しない。
