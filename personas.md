# task-relay ペルソナ定義

文書版: v1.0

参照名方針:
- 本ファイル `personas.md` は freeze された基準文書として扱う。
- 派生文書は `personas.md` を固定参照してよい。
- 将来の大きな改訂は既存 `personas.md` を改名せずに保持したまま、必要なら別名の新文書として追加する。
- したがって、本ファイル内の `文書版` は内容識別のために使い、派生文書の参照パス解決には使わない。

task-relay をペルソナ駆動開発で再定義するための上位文書。
本書は「誰のために、どの状況で、何を失敗させないための道具か」を定義する。
アーキテクチャや状態機械そのものではなく、設計判断の優先順位を固定することを目的とする。

---

## 0. 本書の使い方

- 新機能、運用フロー、UI、通知、権限、障害時挙動を決める際は、まず本書の primary persona に照らして必要性を判断する。
- primary persona の目標達成を阻害する変更は、secondary persona の利便性より優先して避ける。
- 実装詳細で迷った場合は「最もストレスの高い状況にいる primary persona が安全に状況把握できるか」を優先基準とする。
- 本書で定義しない技術詳細は、`basic-design-v1.0.md` と `detailed-design-v1.0.md` に委譲する。

---

## 1. プロダクトの前提

task-relay は、個人開発者が AI agent を使った実装タスクを安全に投入・監視・停止・再開するための task runner である。

このプロダクトが解決すべき中核課題は次の 3 つである。

1. 人間が常時 PC の前にいなくても、AI 実行の状態を見失わないこと。
2. AI が暴走・停止・競合したときに、人間が状況を理解し介入できること。
3. 個人運用の軽さを維持しつつ、破壊的な失敗だけは避けること。

---

## 2. ペルソナ一覧

| ID | 名称 | 優先度 | 概要 |
|---|---|---|---|
| P1 | モバイル併用の個人開発者 | Primary | 日中はスマホ中心、必要時のみ PC で深掘りする 1 人開発者 |
| P2 | 深夜障害対応をする運用者としての自分 | Secondary | 障害時に復旧・停止・再開判断を素早くしたい同一人物 |
| P3 | 変更履歴を後から検証する監査者としての自分 | Secondary | 何が起きたかを後追いで説明したい同一人物 |
| P4 | 継続保守を行うメンテナとしての自分 | Secondary | upgrade、backup、restore、secret 更新を安全に回したい同一人物 |

補足:

- P2 と P3 と P4 は別人ではなく、P1 と同一人物の異なるモードである。
- したがって、マルチユーザー最適化より「1 人が状況ごとに役割を切り替える」前提を優先する。

## 3. Primary Persona

### 3.1 P1: モバイル併用の個人開発者

**プロフィール**

- 1 人で複数の開発タスクを並行して進める。
- 日中はスマートフォンから Discord を確認する時間が多い。
- 深い確認やレビューは Forgejo を PC で開いて行う。
- LLM やエージェントは活用したいが、完全放任は信用していない。
- 速さは欲しいが、リポジトリ破壊や状態不整合は避けたい。

**この人が達成したいこと**

- 思いついた作業をすぐ task として投入したい。
- 移動中でも「今どうなっているか」を短時間で把握したい。
- 問題が起きたら「何が危険で、今なにをすべきか」を迷わず知りたい。
- 自宅や作業 PC に戻ったとき、続きを自然に引き継ぎたい。

**この人の不安**

- AI が勝手に広範囲を変更していないか。
- 障害が起きたのに気づかず、放置で状況が悪化しないか。
- スマホでは詳細が見えず、重要な停止判断を誤らないか。
- 途中停止後に「再開してよい状態か」が分からなくならないか。
- LLM の利用料が気づかないうちに膨らんでいないか。
- rate limit に当たって作業が詰まっていないか、または無自覚に回し続けていないか。

**この人の行動特性**

- まず短い通知を見る。
- 危険時だけ詳細を開く。
- 日常運用では最小入力を好む。
- 危険時は自動化より明示的な停止・承認・再試行を好む。

**この人にとっての成功**

- 重要イベントだけが Discord に来る。
- Discord だけで最低限の投入・停止・承認ができる。
- Forgejo を開けば、履歴と現在地が矛盾なく追える。
- 障害復旧後に「どこから再開すべきか」が明確である。
- 想定外のコスト急増や rate 枯渇には早めに気づける。

**この人にとっての失敗**

- 通知過多で本当に重要なものを見落とす。
- 失敗時に状況把握のため複数箇所を行き来しないといけない。
- task が止まったのか走っているのか分からない。
- 手動介入の余地がない、または介入が危険すぎる。
- 想定外のコスト急増に気づかない。
- rate limit 枯渇に気づかず回し続けてしまう。

## 4. Secondary Personas

### 4.0 Secondary Persona の裁定順

secondary persona 同士の要求が衝突した場合、裁定順は固定する。

1. 障害対応中は P2 を最優先とする。
2. 保守作業中は P4 を次優先とする。
3. 上記 2 つの条件に当てはまらない通常時は P3 を優先する。

補足:

- P2 は「状況悪化を止める」「止めるか待つかを迷わせない」ことを優先する。したがって障害時は、監査上きれいでも判断に時間がかかる設計を採らない。
- P4 は「繰り返し安全に保守できること」を優先する。したがって upgrade、restore、retention、secret rotation の文脈では、監査表現の細かさより手順の再現性を優先する。
- P3 は「後から説明できること」を優先する。したがって安全性と運用性が同等なら、履歴・因果・責務分離がより明確な案を採る。
- ただし、いかなる場合も P1 の目標達成を阻害してはならない。secondary 間の裁定は P1 を満たした後にだけ適用する。

### 4.1 P2: 深夜障害対応をする運用者としての自分

**状況**

- 夜間や外出先で障害通知を受ける。
- 時間をかけずに「止める」「待つ」「再開する」を判断したい。

**最重要ニーズ**

- 障害の種類が局所か全体かを 1 画面で判別できること。
- 現在の in-flight task と停止済み task を 1 画面で区別できること。
- 強制介入コマンドが少数で、意味が明確であること。
- 復旧後の再開ルールが機械的であること。

**設計含意**

- breaker / degrade / cancel / unlock は意味を混ぜない。
- 停止系通知には「何が止まったか」「人間がやるべきこと」を含める。
- 強制コマンドは admin 限定、少数、可逆性の低さを明記する。
- 「1 画面で判別する」surface は Discord の `/status` 応答を主とする。

**この人の行動特性**

- 通知を見た瞬間に、局所か全体かだけをまず判別する。
- 眠いときは長文を読まない。短い action と禁止事項を頼りにする。
- 「いま動くべきか」「朝まで待てるか」を最優先で決める。
- 調査の深掘りは翌朝の自分に渡す。

**この人にとっての成功**

- 通知を見た時点で、局所障害か全体障害かをすぐ判別できる。
- 停止・再開・待機のどれを選ぶべきかが明確である。
- 強制介入コマンドを使う前に、副作用の大きさを理解できる。
- 復旧後に「次は何を確認するか」が機械的に決められる。

**この人にとっての失敗**

- 障害の種類が曖昧で、止めるべきか待つべきか判断できない。
- breaker、degrade、cancel、unlock の意味が重なって見える。
- 深夜対応で複数画面や複数ログを往復しないと状況判断できない。
- 復旧後の再開ルールが曖昧で、再発を招く。

### 4.2 P3: 変更履歴を後から検証する監査者としての自分

**状況**

- 数日後に「なぜこの変更が起きたか」を確認したくなる。
- 障害後に再発防止のため履歴を追う。

**最重要ニーズ**

- task ごとの状態遷移が時系列で追えること。
- AI 実行・人間承認・投影結果が分離して記録されること。
- 真実源と mirror の役割差が明確であること。

**設計含意**

- task truth と operational metadata を混ぜない。
- outbox / inbox / tool log / system event の責務を分離する。
- 監査コメントと current snapshot を分ける。

**この人の行動特性**

- 時系列を辿って因果を再構成する。
- ひとつの表示面だけで結論を出さず、truth と mirror を切り分けて読む。
- 例外ケースや再送も「なぜそうなったか」を説明できるまで掘る。
- 属人的メモではなく、再現可能な手順と記録を好む。

**この人にとっての成功**

- 後日見返しても、task ごとの状態遷移が一貫して追える。
- AI 実行、人間承認、外部反映が混ざらず説明できる。
- 「何が真実で、何が mirror か」を迷わず区別できる。
- 再送や rebuild が起きても、それを履歴として説明できる。

**この人にとっての失敗**

- 状態、監査コメント、投影結果が混ざっていて因果が追えない。
- 真実源と mirror のどちらを信じるべきかが曖昧になる。
- 障害後に「何が起きたか」を再構成できない。
- 後から見たとき、人間の介入理由が読み取れない。

### 4.3 P4: 継続保守を行うメンテナとしての自分

**状況**

- 新しいバージョンへ更新する。
- secret を差し替える。
- restore drill や retention の確認を行う。
- 普段は触らないが、壊れると困る基盤部分を時々保守する。

**最重要ニーズ**

- upgrade 手順が短く、壊してはいけない境界が明確であること。
- backup / restore の成否を、手書きメモに頼らず確認できること。
- retention や rotation が、実運用を壊さずに回ること。
- 「いま触っていいもの」と「不用意に触ると危険なもの」が分かること。

**設計含意**

- runbook と restore drill は一過性の資料ではなく、繰り返し実行できる形で維持する。
- backup / replay / rebuild / retention は task 実行系と責務を分離する。
- secret rotation や upgrade は、真実源を壊さず段階的に実施できる設計を優先する。

**この人の行動特性**

- 保守は計画作業としてまとめて実施する。
- runbook を見ながら進め、前回の記憶には依存しない。
- 危険な境界を先に確認してから操作する。
- 作業後の成功判定が明示されていないと不安になる。

**この人にとっての成功**

- upgrade や restore drill を runbook どおり反復実行できる。
- backup / replay / rebuild / retention の境界が明確で、手順を誤りにくい。
- 保守作業後に、安全に運用継続できるかをシステム状態から確認できる。
- secret rotation や retention 設定変更が task 実行系を壊さない。

**この人にとっての失敗**

- 保守手順が属人的で、システム状態だけでは成否を確認できない。
- restore、rebuild、replay の責務が曖昧で、復旧時に迷う。
- retention や rotation が実行系に副作用を与える。
- upgrade や secret 更新のたびに「壊すかもしれない」不安が強い。

## 5. 反ペルソナ

次の利用者像は、現時点では最適化対象にしない。

### 5.1 チーム開発の複数承認者

- 複数人の並行承認フロー
- 組織監査や職務分掌
- 詳細な RBAC

これは現段階では過剰であり、1 人運用の明快さを損なう。

### 5.2 常時 Web UI を開いている重監視ユーザー

- 監視ダッシュボード常駐
- minute 単位の常時監視
- 高頻度 manual refresh 前提の UI

task-relay は「重要時だけ見る」前提を優先する。

### 5.3 完全自律実行を望むユーザー

- 人間介入なしの無停止自動化
- 多少の誤変更より速度を優先

本プロダクトは安全な介入可能性を優先する。

### 5.4 反ペルソナ再評価の tripwire

以下の兆候が 2 か月連続で観測された場合、反ペルソナを再評価対象に上げる。

- 同種の要望や utterance が 3 件/月以上継続する。
- runbook やレビューで「現行ペルソナでは説明しきれない」運用判断が繰り返し発生する。
- 既存ペルソナを無理に拡張した結果、primary / secondary の裁定が毎回ぶれる。
- 派生ドキュメント側で例外運用が常態化する。

## 6. ジョブ理論

### 6.1 機能的ジョブ

- 思いついた作業を安全に task 化する (P1)
- AI 実装を branch 競合なく進める (P1)
- 実行結果をレビューし、完了か再試行かを決める (P1)
- 異常時に停止・承認・再開・復旧を行う (P2)
- 後から経緯を再構成し、再発防止を決める (P3)
- 基盤を継続保守する (upgrade / restore drill / retention / secret rotation) (P4)

### 6.2 感情的ジョブ

- 「今どこまで進んだか分からない」不安を減らす (P1)
- 「AI が勝手に壊したかもしれない」恐怖を減らす (P1)
- 「スマホからでも制御できる」という安心感を得る (P1)
- 「深夜に状況が勝手に悪化しない」安心を得る (P2)
- 「触っても戻せる」という安心を得る (P4)

### 6.3 社会的ジョブ

- 将来の自分に対して、変更理由を説明できる状態を残す (P3)
- 個人開発でも雑な自動化ではないと自分で納得できる (P1 / P4)

## 7. 代表シナリオ

### 7.1 通常投入

1. P1 は Discord から task を投入する。
2. 短時間で受理結果を知りたい。
3. 詳細は後で Forgejo を見ればよい。

**要求**

- Discord での投入は短い操作で終わること。
- 受理失敗は即わかること。
- 受理後の task は Forgejo で追跡可能であること。

### 7.2 モバイル確認

1. P1 は移動中に通知を見る。
2. 「正常進行」か「介入要」かだけを瞬時に見分けたい。
3. 介入が必要なときのみ詳細を開く。

**要求**

- 通知は絞る。
- 通知文は action-oriented にする。
- 危険時のみ DM を使う。

### 7.3 AI 暴走懸念

1. P1 は AI が広範囲を触ったかもしれないと疑う。
2. すぐ止めたい。
3. 後で PC から状況を精査したい。

**要求**

- `/cancel` と停止伝播が確実であること。
- 停止後に「まだ subprocess が走っているか」を見失わないこと。
- 停止理由と影響範囲が追跡可能であること。

### 7.4 深夜障害

1. P2 は `system_degraded` 通知を受ける。
2. まず局所障害か全体障害かを判断する。
3. 復旧後、どこへ戻すかを迷いたくない。

**要求**

- `human_review_required` と `system_degraded` の意味を厳密に分ける。
- breaker open は task stop ではなく dispatch stop だと分かること。
- 復帰ルールが deterministic であること。

### 7.5 事後監査

1. P3 は後日、task の経緯を確認する。
2. 「誰が何を承認し、何が失敗し、どこまで反映されたか」を追いたい。

**要求**

- state と projection と tool log が分離記録されること。
- mirror が真実源ではないことが明示されること。
- 再送や rebuild が説明可能であること。

### 7.6 継続保守

1. P4 は upgrade や restore drill を実施する。
2. task 実行中の系と、保守のために触る系を混同したくない。
3. 作業後に「今の状態で安全に運用継続できるか」を確認したい。

**要求**

- backup / restore / replay / rebuild の責務境界が明確であること。
- runbook がそのまま作業手順として使えること。
- retention や rotation が task 実行を壊さないこと。

## 8. ペルソナから導く設計原則

### 8.1 通知は少なく、重く

- 通知の量より重要度分離を優先する。
- 進捗通知より介入通知を優先する。

### 8.2 自動化より介入可能性

- 危険時に止められない自動化は採らない。
- 自動再試行は副作用のない範囲に限定する。

### 8.3 1 人運用に最適化する

- 権限、操作、UI は単一運用者前提で単純に保つ。
- 将来の多人数対応のために現在の明快さを犠牲にしない。

### 8.4 真実源と見た目を分ける

- SQLite truth と Forgejo mirror を混同しない。
- 後から見て説明できる状態遷移を優先する。

### 8.5 モバイルでは判定、PC では精査

- Discord は投入、停止、承認、要点確認に限定する。
- 精査や深い履歴確認は Forgejo / log 側に寄せる。

### 8.6 保守作業は反復可能であること

- upgrade、backup、restore、retention は「一度きりの手順」ではなく繰り返し回せる設計を優先する。
- 保守機能は task 実行中の判断系と混ざらないように分離する。

## 9. 機能優先順位

### 9.1 Must Have

- 安全な task 投入 (P1)
- 明確な状態遷移 (P1, P3)
- 停止 / 承認 / 再試行 / 復旧 (P1, P2)
- 監査可能な履歴 (P3)
- モバイルでの要点把握 (P1, P2)
- 反復可能な保守手順 (upgrade / restore / retention) (P4)

### 9.2 Should Have

- 復旧時の deterministic な resume (P2)
- rebuild と projection の冪等性 (P3)
- 低ノイズな通知設計 (P1, P2)
- 局所障害と全体障害を 1 画面で判別する運用ビュー (P2)
- restore drill を runbook どおり反復実行できる環境 (P4)
- コスト / rate の現況把握と通知設計 (P1, TBD)

コスト / rate TBD の解除 tripwire:

- source of truth は `docs/guides/persona-metrics.md` §2 とする。
- `personas.md` では「P1 向け cost / rate 可視化は tripwire 到達時に正式着手対象へ上げる」という方針だけを保持する。

### 9.3 Won't Optimize Yet

- 多人数ワークフロー
- 複雑な承認フロー
- 常時監視ダッシュボード
- エンタープライズ向け RBAC

## 10. 評価質問

新しい設計や機能を入れる前に、最低限次を確認する。

1. P1 はスマホだけで「問題の有無」を判断できるか。
2. P1 向けのコスト / rate 可視化を入れる場合、その threshold、集計 window、通知頻度、通知先が明示されているか。
3. P2 は障害時に「止める・待つ・再開する」を迷わず選べるか。
4. P2 は局所障害と全体障害を 1 画面で判別できるか。
5. P3 は後から状態遷移、介入理由、および truth / mirror 境界を説明できるか。
6. P4 はこの変更後も runbook どおりに upgrade / restore / retention を回せるか。
7. この変更は 1 人運用の単純さを壊していないか。
